二、实验内容及其完成情况

为了比较SQL查询中不同连接算法（Nested Loop Join, Hash Join, Merge Sort Join）的作用，我们可以设计一组实验，分析在不同条件下（如大小表、有无索引、不同连接条件）这些连接算法的表现。以下是设计实验的详细步骤和说明：

1. 实验设计

数据库和表结构选择：

我们选择一个包含多个表的数据库，并设计适当的表结构。为了实验的通用性，假设我们使用一个简单的订单管理数据库，包括如下两个表：

  • Customers 表：包含顾客信息。

    CREATE TABLE Customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    city VARCHAR(100)
);

  • Orders 表：包含顾客订单信息。

    CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    total_amount DECIMAL(10, 2)
);

数据量设置：

  • 小表：Customers表只有100条记录。
  • 大表：Orders表包含1000条记录。

实验目的：

  1. 比较三种连接算法（Nested Loop Join, Hash Join, Merge Sort Join）的性能： 主要评估在不同数据量和不同条件下的执行时间、存取路径和策略。
  2. 考虑索引影响： 分析当连接条件上有索引时和没有索引时，三种连接算法的表现差异。
  3. 比较不同连接条件的影响： 分析当连接条件不同（例如，单一条件 vs 多条件）时的执行时间差异。

2. 实验步骤

2.1 创建数据并插入数据

我们首先在数据库中创建并插入测试数据。采用循环插入。

DO $$ 
DECLARE 
    i INT := 1;
BEGIN
    WHILE i <= 100 LOOP
        INSERT INTO Customers (customer_id, customer_name, city)
        VALUES (i, 'Customer ' || i, 'City ' || i);
        i := i + 1;
    END LOOP;
END $$;


DO $$ 
DECLARE 
    i INT := 1;
BEGIN
    WHILE i <= 1000 LOOP
        INSERT INTO Orders (order_id, customer_id, order_date, total_amount)
        VALUES (
            i,  -- Order ID
            (i % 100) + 1,  -- Customer ID, 假设有100个顾客
            '2023-10-' || LPAD((i % 30) + 1, 2, '0'),  -- 随机生成日期
            round(random() * 500, 2)  -- 随机生成订单金额（0到500之间）
        );
        i := i + 1;
    END LOOP;
END $$;

2.2 创建索引

实验中，我们考虑在连接条件上创建索引，测试不同情况的效果。

-- 在Orders表的customer_id列上创建索引
CREATE INDEX idx_customer_id ON Orders(customer_id);

2.3 执行不同的连接查询

我们使用三种连接算法（Nested Loop Join, Hash Join, Merge Sort Join）执行如下SQL查询，并分别测试它们的执行时间：

-- 使用 Nested Loop Join
EXPLAIN ANALYZE
SELECT *
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id;

-- 使用 Hash Join
EXPLAIN ANALYZE
SELECT *
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id;

-- 使用 Merge Sort Join
EXPLAIN ANALYZE
SELECT *
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id;

在EXPLAIN ANALYZE的输出中，我们可以看到每种连接的执行计划，包括：

  • 连接类型： 是否是嵌套循环连接、哈希连接还是归并连接。
  • 访问路径： 查询使用的是全表扫描、索引扫描，还是其他优化路径。
  • 时间代价： 每一步操作的时间代价。
  • 行数估算： 各个阶段的行数估算。

3. 结果分析

3.1 执行计划分析

EXPLAIN ANALYZE的输出会告诉我们数据库选择了哪个连接算法及其执行路径。对于每种连接算法，我们可以通过以下指标来进行比较：

  • 时间代价（Time Cost）： 表示执行查询所需的时间，通常嵌套循环连接的代价较高，尤其是在没有索引的情况下。
  • IO操作： 如果选择了全表扫描或大规模的磁盘访问，代价将较高。
  • 并行度： 数据库是否能够通过并行化操作加速某些连接操作（例如哈希连接）。

3.2 时间对比

不同的连接算法在不同的情况下一定会表现出不同的时间开销：

  • Nested Loop Join（嵌套循环连接）： 如果没有索引，嵌套循环连接会对每个外层行进行内层查找，时间复杂度是O(N * M)，通常在小表和有索引的情况下表现较好。
  • Hash Join（哈希连接）： 当没有合适的索引时，哈希连接通常表现较好，因为它将数据分成多个桶并进行散列，但对内存的需求较高。
  • Merge Sort Join（归并排序连接）： 归并连接通常在两张表都已排序时效果最好，因此在有索引的情况下效果更好。

3.3 存取路径和策略

  • 无索引时： 查询往往会选择全表扫描（Seq Scan），连接操作可能使用嵌套循环连接（Nested Loop），这会导致较高的时间代价。
  • 有索引时： 查询优化器可能选择索引扫描（Index Scan），此时哈希连接或归并连接的表现较好。

4. 实验总结

通过上述实验，我们可以得出结论：

  1. 无索引时， 嵌套循环连接可能会因为大量的全表扫描而效率低下。
  2. 有索引时， 哈希连接和归并排序连接可能会由于使用索引扫描而表现较好，尤其是在大表中。
  3. 连接条件的复杂性（例如多条件连接）也会影响执行计划的选择，通常需要通过EXPLAIN来进一步调整优化。
