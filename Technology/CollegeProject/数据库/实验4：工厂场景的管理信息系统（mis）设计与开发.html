二、实验内容及其完成情况

为了实现一个基于PostgreSQL的MIS（管理信息系统）功能表设计，我们首先需要设计一个数据库模型来存储零件、供应商、仓库、职工等信息。然后，我们可以通过Python与PostgreSQL进行交互，实现相应的功能。以下是系统的功能表设计和相关实现思路。

1. MIS 功能设计

基于上述数据库表结构，MIS系统可以包括以下基本功能模块：

  1. 零件管理
    • 添加、删除、修改零件信息
    • 查询零件信息（按编号、名称、类型等）
  2. 供应商管理
    • 添加、删除、修改供应商信息
    • 查询供应商信息（按供应商名称、地址等）
  3. 仓库管理
    • 添加、删除、修改仓库信息
    • 查询仓库信息（按仓库名称、地址等）
  4. 职工管理
    • 添加、删除、修改职工信息

    • 添加进货记录（零件、数量、供应商）

  6. 库存管理
    • 查询某仓库的零件库存
    • 更新零件库存

2. 数据库表设计

首先，设计数据库表结构。以下是一些基本的表结构：

2.1 零件表（Parts）

用于存储零件的基本信息。

CREATE TABLE parts (
    part_id SERIAL PRIMARY KEY,  -- 零件编号
    part_name VARCHAR(100) NOT NULL,  -- 零件名称
    price DECIMAL(10, 2) NOT NULL,  -- 单价
    part_type VARCHAR(50),  -- 零件类型
    supplier_id INT REFERENCES suppliers(supplier_id)  -- 关联供应商
);

2.2 供应商表（Suppliers）

用于存储供应商信息。

CREATE TABLE suppliers (
    supplier_id SERIAL PRIMARY KEY,  -- 供应商编号
    supplier_name VARCHAR(100) NOT NULL,  -- 供应商名称
    address VARCHAR(255),  -- 地址
    phone VARCHAR(20)  -- 电话
);

2.3 仓库表（Warehouses）

用于存储仓库信息。

CREATE TABLE warehouses (
    warehouse_id SERIAL PRIMARY KEY,  -- 仓库编号
    warehouse_name VARCHAR(100) NOT NULL,  -- 仓库名称
    address VARCHAR(255)  -- 仓库地址
);

2.4 职工表（Employees）

用于存储职工信息。

CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,  -- 职工编号
    name VARCHAR(100) NOT NULL,  -- 职工姓名
    gender VARCHAR(10),  -- 性别
    hire_date DATE NOT NULL,  -- 进厂时间
    title VARCHAR(50),  -- 职称
    warehouse_id INT REFERENCES warehouses(warehouse_id),  -- 关联仓库
    is_manager BOOLEAN DEFAULT FALSE  -- 是否为仓库组长
);

2.5 批发进货表（Purchases）

用于记录批发进货信息。

CREATE TABLE purchases (
    purchase_id SERIAL PRIMARY KEY,  -- 进货编号
    part_id INT REFERENCES parts(part_id),  -- 关联零件
    supplier_id INT REFERENCES suppliers(supplier_id),  -- 关联供应商
    quantity INT NOT NULL,  -- 进货量
    purchase_date DATE NOT NULL  -- 进货日期
);

2.6 仓库零件库存表（Warehouse_Stock）

用于记录仓库中的零件库存。

CREATE TABLE warehouse_stock (
    stock_id SERIAL PRIMARY KEY,  -- 库存记录编号
    warehouse_id INT REFERENCES warehouses(warehouse_id),  -- 关联仓库
    part_id INT REFERENCES parts(part_id),  -- 关联零件
    quantity INT NOT NULL  -- 库存数量
);

2.7数据库表和用户角色管理表

在数据库中设置一个用户表，用来存储用户名、密码和角色信息。你可以使用一个简单的用户表 users，它包含以下字段：

CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    password VARCHAR(50) NOT NULL,
    role VARCHAR(20) NOT NULL -- 'manager' 或 'staff'
);

# User login check
def login(username, password):
    conn = connect_db()
    if conn:
        cursor = conn.cursor()
        cursor.execute("SELECT role FROM users WHERE username = %s AND password = %s", (username, password))
        result = cursor.fetchone()
        cursor.close()
        conn.close()
        if result:
            return result[0]  # Return the role: 'manager' or 'staff'
        else:
            return None
    return None

三、实验总结

当然，总结实验中遇到的问题及其解决思路是非常重要的，这有助于改进系统并提高未来项目的成功率。以下是两个示例问题的总结：

问题1：数据导入时出现重复记录

1. 问题描述

在将供应商数据导入数据库时，发现有些供应商记录出现了重复，导致数据冗余和不一致。

2. 问题分析（可能的原因、难点、挑战）

  • 原因：
    • 数据源中存在重复记录。
    • 导入过程中没有进行唯一性校验。
    • 数据清洗不彻底。
  • 难点：
    • 如何高效地检测和删除重复记录。
    • 如何在导入过程中自动防止重复记录的插入。
  • 挑战：
    • 数据量较大时，性能问题可能会成为一个瓶颈。
    • 需要确保数据的一致性和完整性。

3. 解决方案

  • 数据清洗：
    • 在导入前对数据源进行预处理，使用脚本或工具删除重复记录。
    • 使用SQL查询检测和删除重复记录，例如： 
      DELETE FROM Suppliers
WHERE supplier_id NOT IN (
    SELECT MIN(id)
    FROM Suppliers
    GROUP BY supplier_name, address, phone
);
  • 唯一性约束：
    • 在数据库表中添加唯一性约束，确保关键字段的唯一性。 
      ALTER TABLE Suppliers ADD CONSTRAINT unique_supplier UNIQUE (supplier_name, address, phone);
  • 批量导入时的校验：
    • 在导入数据时，先查询数据库中是否存在相同的记录，如果不存在再插入新记录。
    • 使用事务管理，确保数据的一致性。

问题2：库存管理中的数据同步问题

1. 问题描述

在多仓库环境中，不同仓库的库存数据同步不及时，导致库存信息不一致，影响生产计划和销售订单的处理。

2. 问题分析（可能的原因、难点、挑战）

  • 原因：
    • 数据同步机制不完善。
    • 网络延迟或中断导致数据同步失败。
    • 缺乏实时监控和报警机制。
  • 难点：
    • 实现高效的实时数据同步。
    • 处理网络不稳定带来的数据丢失问题。
  • 挑战：
    • 确保数据同步的可靠性和一致性。
    • 提高系统的容错能力。

3. 解决方案

  • 实时同步机制：
    • 使用消息队列（如RabbitMQ、Kafka）实现库存数据的实时同步。
    • 在每个仓库的库存变化时，发布一条消息到消息队列，其他仓库订阅该消息并更新自己的库存数据。
  • 数据校验和补偿：
    • 定期进行数据校验，确保各仓库的库存数据一致。
    • 如果发现数据不一致，使用补偿机制进行修复。
  • 监控和报警：
    • 实施实时监控系统，监控数据同步的状态和性能。
    • 设置报警机制，当数据同步失败或延迟超过阈值时，发送警报通知相关人员进行处理。

问题3：性能优化问题

1. 问题描述

在系统运行过程中，随着数据量的增加，查询和操作的响应时间逐渐变慢，影响用户体验。

2. 问题分析（可能的原因、难点、挑战）

  • 原因：
    • 查询语句不够优化。
    • 索引设计不合理。
    • 硬件资源不足。
  • 难点：
    • 如何高效地优化查询语句。
    • 如何选择合适的索引。
  • 挑战：
    • 在不影响系统正常运行的情况下进行性能优化。

3. 解决方案（思路和方向）

  • 查询优化：
    • 使用EXPLAIN分析查询语句，找出性能瓶颈。
    • 优化查询逻辑，减少不必要的数据扫描。
  • 索引优化：
    • 根据查询频率和条件，合理添加索引。
    • 定期检查和优化索引，避免过度索引导致的性能下降。
  • 硬件升级：
    • 增加内存和CPU资源，提高系统处理能力。
    • 使用更快的存储介质，如SSD。
  • 分布式架构：
    • 考虑使用分布式数据库或缓存系统（如Redis），分散数据访问压力。
    • 实现负载均衡，提高系统的整体性能。
